# Stock Signal Backtest

## Goal

Backtest v1 verifies whether stored Quant Signal scores had useful forward-return behavior.

The app does not treat Signal as a prediction. It stores signal snapshots, waits for future daily candles, and then measures realized returns after fixed trading-day horizons.

## Tables

### stock_signal_snapshot

`stock_signal_snapshot` stores one daily row per symbol and signal version.

It is written by `StockSignalRefreshService` after `stock_signal_latest` is refreshed. This preserves the exact Signal score and factor scores that were visible on that signal date.

`snapshot_mode` separates real live snapshots from reconstructed historical snapshots:

- `LIVE`: saved by the normal Signal refresh flow.
- `HISTORICAL_RECONSTRUCTED`: generated by the historical backtest seed batch.

Important columns:

- `symbol`
- `signal_date`
- `snapshot_mode`
- `calculated_at`
- `signal_version`
- `integrated_score`
- `valuation_score`
- `quality_score`
- `growth_score`
- `stability_score`
- `earnings_score`
- `analyst_score`
- `news_score`
- `momentum_score`
- `risk_score`
- `institution_score`
- `data_quality_score`
- `reasons_json`, `cards_json`, `source_freshness_json`, `raw_json`

### stock_signal_backtest_result

`stock_signal_backtest_result` stores realized forward returns for a snapshot.

Backtest v1 uses these horizons:

- 5 trading days
- 20 trading days
- 60 trading days

Entry is the first candle strictly after `signal_date`. Exit is the Nth future trading-day close for the selected horizon. This avoids using the same candle close that may already have informed the signal.

### stock_signal_weight_profile

`stock_signal_weight_profile` stores named Signal weight profiles.

Initial profiles:

- `DEFAULT`: the current production reference profile.
- `BACKTEST_V1`: a diagnostics-generated candidate profile. It is inserted with `active=false`.
- `BACKTEST_V2`: a diagnostics-generated candidate profile with orthogonalization and correlation penalties. It is inserted with `active=false`.

`active=false` means the profile is not applied to live `/stocks` Signal. Backtest uses it only for comparison.

### stock_signal_weight_profile_item

`stock_signal_weight_profile_item` stores one factor weight per profile.

The factor set is:

- Valuation
- Quality
- Growth
- Stability
- Earnings
- Analyst
- News
- Momentum
- Risk
- Institution

## Services

### StockSignalRefreshService

Signal refresh flow:

1. Build current Signal from stored source data.
2. Apply factor ranking and guardrails.
3. Save `stock_signal_latest`.
4. Save `stock_signal_snapshot`.
5. Ask `StockBacktestService` to fill any completed forward-return rows.

### HistoricalBacktestSeedService

`HistoricalBacktestSeedService` creates `HISTORICAL_RECONSTRUCTED` snapshots so the app can validate Signal behavior without waiting months for new `LIVE` snapshots.

The historical seed intentionally avoids known data leakage:

- Momentum and Risk use only `stock_candle_daily` rows where `trade_date <= signal_date`.
- SEC financials use only `stock_sec_financial_standard` rows where `filed_at <= signal_date`.
- News uses `published_at` in the 30 days before the signal date.
- Analyst trend uses the latest `period_date <= signal_date`.
- Institution uses collected 13F quarterly flow only when `report_quarter + 45 days <= signal_date`.
- Finnhub latest metric snapshots are not used for historical reconstructed Signal.

Default seed frequency is monthly. The current batch chooses historical month-end trading dates far enough before the latest candle date so 5D/20D/60D forward returns can be calculated immediately.

Run:

```cmd
run-historical-backtest-seed.cmd
```

Equivalent Java options:

```cmd
java -Dspring.profiles.active=mariadb -jar target\codex-demo-0.0.1-SNAPSHOT.jar --spring.main.web-application-type=none --app.signal.refresh.enabled=false --app.batch.historical-backtest.enabled=true --app.batch.historical-backtest.exit-on-complete=true
```

### StockBacktestService

`StockBacktestService` owns:

- Candle coverage audit
- Pending snapshot detection
- Result refresh when a snapshot was recalculated with newer factor scores
- 5D/20D/60D forward-return calculation
- Signal bucket summaries
- Top 10% vs bottom 10% summaries
- Factor high/low return comparison
- Factor performance diagnostics
- Sector-neutral factor checks
- Year stability checks
- Weight candidate scoring
- Weight profile persistence
- DEFAULT vs BACKTEST_V1/BACKTEST_V2 profile comparison
- Walk-forward profile validation
- Sector bias summaries

The `/signals/backtest` page is a validation surface. It may refresh completed result rows from already stored snapshots and candles, but it does not recalculate live Signal scores.

Snapshot/result summaries are filtered by the requested index's current constituents. The current menu route uses `SP500`, so Nasdaq 100 or Dow 30-only symbols do not leak into the S&P 500 validation metrics.

### StockPortfolioBacktestService

`StockPortfolioBacktestService` turns completed per-symbol backtest rows into investable portfolio simulations.

Portfolio v1/v2 does not create a new persistence table. It computes the page from stored `stock_signal_backtest_result` rows, so it is deterministic for the same stored input data.

Portfolio v1 strategy grid:

- Holding horizons: 20D and 60D
- Portfolio sizes: Top 10, Top 20, Top 50
- Weight methods: equal weight, Signal weight, market-cap adjusted weight
- Risk modes: Base and Risk controlled
- Transaction cost: dynamic traded-notional cost
- Benchmark: same-date S&P 500 current-constituent market-cap weighted proxy

Risk Model v1 adds a second portfolio construction path without changing live Signal:

- `Base` keeps the original Top N selection and weighting.
- `Risk controlled` applies a trailing-risk adjusted candidate score.
- Sector concentration is limited with a 30% target cap.
- Position concentration is limited with an 8% cap, or 12% for Top 10 portfolios.
- Trailing volatility uses 126 daily returns from `stock_candle_daily`.
- Liquidity uses trailing 63D average dollar volume.
- Beta uses trailing 126D covariance against an equal-weight universe return proxy.

Risk-controlled portfolios may hold implicit cash when all selected names or sectors hit their caps. This is intentional in v1 because forcing 100% investment can break the risk constraints.

Portfolio Backtest v2 adds a more realistic cost and comparison layer:

- Transaction cost is charged against actual changed position notional, not only a fixed turnover scalar.
- The base cost is 0.04% per traded notional.
- Liquidity impact increases cost when trailing 63D average dollar volume is low.
- Volatility impact increases cost when trailing 126D volatility is high.
- Beta impact increases cost for high-beta names.
- `Risk Impact Review` pairs every Risk-controlled strategy with the matching Base strategy.
- The review compares Sharpe delta, MDD delta, beta delta, sector concentration delta, cost delta, and excess return delta.
- Positive MDD delta means drawdown improved because the Risk-controlled MDD is less negative than Base.

Metrics:

- Cumulative return
- Benchmark return
- Excess return
- Annualized return
- Annualized volatility
- Sharpe ratio with risk-free rate set to 0
- Maximum drawdown
- Benchmark beat rate
- Average turnover
- Average transaction cost
- Average maximum sector weight
- Average maximum position weight
- Average portfolio beta
- Average trailing portfolio volatility
- Average trailing dollar-volume liquidity

Limitations:

- Benchmark market cap uses current `company_profile.market_cap`, not point-in-time market cap.
- Risk beta uses an equal-weight current-universe proxy because point-in-time S&P 500 index returns are not stored yet.
- Liquidity is a dollar-volume proxy, not an executable volume or spread model.
- Transaction cost is still a proxy; it does not model bid/ask spread, intraday volume participation, market impact curves, borrow cost, or tax.
- Rebalance dates are historical snapshot dates, currently monthly in the seed.
- Weekly rebalance testing needs weekly historical snapshot seed.
- Portfolio v1 uses stored Signal scores and does not promote or change live weights.

## Screens

### /signals/backtest

`/signals/backtest` shows:

- Candle coverage: how many symbols have enough daily candles for 5D/20D/60D tests
- Signal buckets: average forward returns by Signal range
- Top vs bottom: top 10% score group versus bottom 10%
- Factor validation: high factor score versus low factor score
- Factor diagnostics: sample count, average return, hit rate, IC, and High 70+ vs Low 40- spread for each factor
- Sector neutral check: factor spread by sector so a factor is not promoted only because one sector dominated the result
- Year stability: factor spread by year so one good year does not drive the entire decision
- Weight candidates: a non-binding weight action candidate based on 20D spread, 60D spread, 20D IC, and year stability
- Profile A/B Backtest: compares stored `DEFAULT` Signal against computed `BACKTEST_V1` and `BACKTEST_V2` scores
- Walk-forward Backtest: trains temporary profiles only on years before the test year, then validates them on the next year
- Weight Profile Items: shows factor weights saved for each profile
- Sector bias: whether a result is concentrated in a sector

### /signals/portfolio

`/signals/portfolio` shows:

- Best strategy summary cards
- Risk Model v1 constraint and input summary
- Risk Impact Review for matching Base versus Risk-controlled strategy pairs
- Full Top N / horizon / weighting comparison table
- Recent rebalance periods for the selected default strategy
- Sector concentration table
- Latest selected positions
- Validation notes and current benchmark assumptions

## Weight Tuning

Backtest v1 does not automatically change live Signal weights.

The Factor Performance Diagnostics layer is a review gate before any automatic weighting:

- `Factor diagnostics` measures whether each factor has positive forward-return behavior.
- `Sector neutral check` measures whether the factor works across sectors instead of only reflecting a sector bet.
- `Year stability` measures whether the factor is repeatable across years.
- `Weight candidates` labels factors as increase candidate, observe/hold, or reduce/reverse candidate.

These labels are advisory only. They do not update `stock_signal_latest`, `stock_signal_snapshot`, or any production Signal calculation.

Weight Profile v1 adds a second review gate:

- `DEFAULT` is kept as the live reference profile.
- `BACKTEST_V1` is generated from diagnostics and saved to DB with `active=false`.
- `BACKTEST_V2` starts from the v1 candidate, then applies sector/size neutral scoring and factor-correlation penalties. It is also saved with `active=false`.
- `/signals/backtest` computes hypothetical `BACKTEST_V1` and `BACKTEST_V2` scores from stored factor scores.
- The page compares `DEFAULT`, `BACKTEST_V1`, and `BACKTEST_V2` by 5D/20D/60D average return, hit rate, IC, and Top 10% vs Bottom 10% spread.
- This comparison still does not update `/stocks` or `/stocks/{symbol}` Signal.

Walk-forward validation adds a third review gate:

- For each test year, the app uses only snapshots before that year to generate temporary walk-forward profiles.
- The temporary profile is then evaluated on the held-out test year for 5D/20D/60D returns.
- Walk-forward profiles are not persisted to `stock_signal_weight_profile`; they are validation-only profiles.
- A profile should not be promoted if it only improves the in-sample `Profile A/B Backtest` but fails walk-forward IC, hit rate, or Top/Bottom spread.

`Weight Promotion Gate` summarizes the promotion decision:

- The gate uses 20D and 60D rows only. 5D remains visible, but it is too noisy for promotion.
- A candidate is an approval candidate only when sample size is large enough, candidate Top/Bottom spread is positive, IC is non-negative, spread delta is at least `+0.30%p`, IC delta is at least `+0.005`, and top-decile hit rate does not deteriorate.
- A candidate is rejected when spread or IC is clearly negative, or spread delta is at least `-0.30%p` worse than `DEFAULT`.
- All other cases are hold.
- The gate does not activate a profile. Production `/stocks` and `/stocks/{symbol}` continue to use stored live `DEFAULT` Signal until a profile is explicitly promoted.

`BACKTEST_V2` adds a first orthogonalization layer:

- The score is converted to a neutralized z-score inside the same sector and current market-cap bucket when enough peers exist.
- If the sector/size peer group is too small, it falls back to sector-only neutralization.
- Highly correlated factor pairs are penalized on the weaker-evidence side, so the profile does not double-count the same style exposure.
- Current `company_profile.market_cap` is used only as a practical size bucket for diagnostics. A stricter future version should store point-in-time market cap snapshots before relying on size-neutral backtests for promotion.

## Current Limitations

Historical seed v1 still has limits:

- Historical index membership is not point-in-time yet; current `index_constituent` membership is used.
- Historical shares outstanding is not available, so historical valuation uses close/EPS rather than market-cap based multiples.
- News and Analyst history depends on what has already been collected in DB.
- Historical Institution factor coverage depends on collected `stock_institution_flow_quarterly` rows.
- Reconstructed snapshots are validation inputs, not values that were truly displayed to users on those past dates.

Portfolio v1 still has limits:

- Benchmark market cap is current, not point-in-time.
- Turnover assumes full rebalance on snapshot dates.
- There is no cash position, risk-free asset, or stop-loss logic.
- There is no portfolio-level beta, liquidity, or borrow-cost model yet.

## Intended Next Step

The next durable Quant AI step is point-in-time risk modeling:

1. Store point-in-time market cap and sector snapshots.
2. Add beta, liquidity, and volatility risk model tables.
3. Run portfolio backtest with sector caps and position caps.
4. Compare unconstrained versus risk-constrained portfolios.
5. Promote weight profiles only when they improve both stock-level diagnostics and portfolio-level outcomes.
